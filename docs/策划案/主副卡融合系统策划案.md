# 废土元年 — 主副卡融合系统策划案

> **版本**：v2.0
> **版本日期**：2026-04-07
> **用途**：定义主副卡融合系统的完整规则，替代原"4合1随机合成"
> **关联文档**：抽卡与合成系统策划案（本文档替代其中"三、合成系统"部分）、融合姬养成系统策划案、职业能力树设计文档、数值配表指导文档、改造晶片打书系统策划案（晶片系统已独立为单独文档）
> **v2.0 变更说明**：移除改造晶片在融合中的参与；随机系数范围从 0.4~0.6 调整为 0.2~0.8；新增副卡选择策略指南；新增能力继承规则（按稀有度分级）；新增同元素加成机制。

---

## 一、系统概述

### 1.1 核心概念

主副卡融合是融合姬品质提升和能力变异的核心途径。玩家选择一张"主卡"和一张"同品质副卡"进行融合，主卡保留身份但数值和能力发生变化，小概率品质提升。融合仅消耗菌丝，不涉及任何改造晶片（改造晶片已独立为"打书"系统，详见《改造晶片打书系统策划案》）。

```
旧系统（已废弃）：4张同品质卡 + 菌丝 → 随机高一品质卡（身份丢失）
新系统：1张主卡 + 1张同品质副卡 + 菌丝 → 主卡进化（身份保留）
```

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| 身份保留 | 主卡的卡牌ID、技能列表、技能等级全部保留 |
| 纯粹融合 | 融合仅涉及主卡、副卡和菌丝，不涉及改造晶片 |
| 风险与收益并存 | 数值加成有较大随机性（0.2~0.8），鼓励策略选择和副卡挑选 |
| 每卡限升1次品质 | 品质+1只有20%概率，且每张卡最多触发1次，不会快速消耗内容 |
| 能力变异 | 能力组合可能触发隐藏融合能力，鼓励实验和探索 |
| 副卡消失 | 融合后副卡消耗消失，不可恢复 |
| 副卡策略性 | 副卡的选择直接影响融合结果，是系统的核心策略深度所在 |

### 1.3 与改造晶片系统的关系

改造晶片（打书）系统已从融合系统中**完全剥离**，成为独立系统。两个系统的关系如下：

| 维度 | 融合系统 | 打书系统（改造晶片） |
|------|---------|-------------------|
| 目的 | 品质提升 + 数值变异 + 能力变异 | 精确修改/替换指定能力 |
| 材料 | 主卡 + 同品质副卡 + 菌丝 | 融合姬卡 + 改造晶片 |
| 随机性 | 高（数值大范围随机，能力随机变化） | 低（指定槽位替换） |
| 适用场景 | 整体强化、品质突破 | 精细化调整、洗技能 |

## 二、融合基础规则

### 2.1 融合条件

| 条件 | 说明 |
|------|------|
| 主卡 | 任意已拥有的融合姬卡 |
| 副卡 | 必须与主卡**同品质** |
| 菌丝 | 消耗菌丝，数量由主卡品质决定 |
| 品质上限 | 主卡品质已达最高（LE）时不可融合 |
| 品质提升限制 | 每张主卡**最多品质+1一次**，已升过品质的卡不可再次触发品质提升 |

### 2.2 不可融合的情况

| 情况 | 说明 |
|------|------|
| 主卡为LE品质 | 已是最高品质，无法再提升 |
| 主副卡品质不同 | 副卡必须与主卡同品质 |
| 菌丝不足 | 融合消耗菌丝，不足时无法进行 |
| 主卡已品质+1 | 该卡已经触发过品质提升，不再有概率升品质（但仍可融合改变数值/能力） |

### 2.3 融合消耗

融合仅消耗菌丝，不涉及任何其他材料：

| 主卡品质 | 菌丝消耗 |
|---------|---------|
| N | 500菌丝 |
| R | 2,000菌丝 |
| SR | 8,000菌丝 |
| SSR | 30,000菌丝 |
| UR | 100,000菌丝 |
| LE | 不可融合 |

## 三、品质提升规则

### 3.1 品质变化概率

| 结果 | 概率 | 说明 |
|------|------|------|
| 品质不变 | 80% | 主卡品质不变，数值和能力仍会变化 |
| 品质+1 | 20% | 主卡品质提升一级（N→R→SR→SSR→UR→LE） |

### 3.2 同元素品质提升加成

如果主卡和副卡**属性相同**（如火+火、水+水），品质提升概率额外增加：

| 条件 | 品质+1概率 |
|------|-----------|
| 主副卡不同元素 | 20% |
| 主副卡相同元素 | **25%**（+5%加成） |

> **设计意图**：同元素加成鼓励玩家选择同属性副卡，增加策略维度。但加成幅度不大，不会让不同元素副卡完全失去价值。

### 3.3 品质提升限制

| 规则 | 说明 |
|------|------|
| 每卡限1次 | 一张卡最多触发品质+1一次，标记 `hasQualityUpgraded = true` |
| 不可逆 | 品质提升后不可降级 |
| LE不可融合 | LE品质为最高品质，不可再融合 |

### 3.4 品质提升后的变化

品质+1时，除了品质标签变化外，还影响以下内容：

| 维度 | 变化 |
|------|------|
| 能力槽上限 | 按新品质的能力槽数量（如SR 4槽→SSR 4槽，但可抽核心能力） |
| 品质系数 | 升级消耗的品质系数按新品质计算 |
| 卡框外观 | 切换为新品质对应的卡框 |
| 能力池 | 后续融合时，能力判定基于新品质的能力池 |

## 四、数值变化规则

### 4.1 核心公式

融合后主卡的数值**重置为1级**，新基础值由以下公式决定：

```
新baseHp = (主卡原始baseHp + 副卡原始baseHp) × 随机(0.2, 0.8)
新baseAtk = (主卡原始baseAtk + 副卡原始baseAtk) × 随机(0.2, 0.8)
新baseSpd = (主卡原始baseSpd + 副卡原始baseSpd) × 随机(0.2, 0.8)
```

**注意**：此处的"主卡原始baseHp"和"副卡原始baseHp"指的是配置表中该卡牌ID的1级初始值，**不是当前等级的数值**。三个属性的随机系数**独立生成**，可能出现HP运气好但ATK运气差的情况。

### 4.2 数值变化分析

以两张SSR坦克为例：

| 情况 | 主卡baseHp | 副卡baseHp | 随机系数 | 新baseHp | 变化 |
|------|-----------|-----------|---------|---------|------|
| 极好 | 3000 | 2800 | 0.8 | 4,640 | +1,640 ↑↑ |
| 较好 | 3000 | 2800 | 0.6 | 3,480 | +480 ↑ |
| 中等 | 3000 | 2800 | 0.5 | 2,900 | -100 ↓ |
| 较差 | 3000 | 2800 | 0.4 | 2,320 | -680 ↓↓ |
| 极差 | 3000 | 2800 | 0.2 | 1,160 | -1,840 ↓↓↓ |

**设计意图**：
- 0.2~0.8的随机范围意味着**期望值为0.5**，即两张卡初始值之和的一半
- **高风险高回报**：运气好可以大幅提升（最高+54%），运气差也会大幅下降（最低-61%）
- 如果副卡初始值与主卡接近，期望结果接近主卡原始值（不亏不赚）
- 如果副卡初始值高于主卡，期望结果高于主卡原始值（有收益）
- 如果副卡初始值低于主卡，期望结果低于主卡原始值（有风险）
- 宽随机范围让每次融合都有强烈的紧张感和惊喜感
- 三个属性独立随机，可能出现"偏科"结果（如HP大增但ATK下降），增加策略趣味

### 4.3 成长系数

融合后主卡的成长系数（hpGrowth / atkGrowth / spdGrowth）**不变**，仍使用主卡原始的成长系数。

### 4.4 品质提升时的数值额外加成

如果融合时触发了品质+1，数值公式不变，但**品质提升本身会带来额外的能力槽扩展**（见第五节），不直接改变数值。

## 五、能力变化规则

### 5.1 能力数量变化

融合后主卡的能力数量按以下规则确定：

```
新能力数量 = 原主卡能力数量 + 随机(-1, 0, +1)
```

| 概率 | 结果 | 说明 |
|------|------|------|
| 33% | -1 | 减少1个能力 |
| 34% | 0 | 能力数量不变 |
| 33% | +1 | 增加1个能力 |

**约束条件**：
- 新能力数量不能低于1
- 新能力数量不能超过当前品质的槽上限
- 如果触发品质+1，槽上限按新品质计算

### 5.2 能力保留判定

确定新能力数量后，按以下优先级决定保留哪些能力：

#### 第一步：必带能力（100%保留）

主卡的 `forcedAbilities` 中配置的能力**必定保留**，不参与随机判定。

#### 第二步：共有能力（大概率保留）

主卡和副卡**都有的能力**，80%概率保留。

#### 第三步：单方能力（按继承规则判定）

只有主卡或只有副卡有的能力，按**能力继承规则**判定（见5.2.1节）。

#### 第四步：补充新能力

如果保留的能力数量少于"新能力数量"，从以下来源补充：

| 来源 | 优先级 | 说明 |
|------|--------|------|
| 副卡独有能力（未被保留的） | 高 | 优先填入副卡的能力 |
| 品质对应能力池 | 中 | 从随机能力池中按权重选取 |
| 隐藏融合能力 | 低 | 特定组合触发（见5.3节） |

#### 第五步：裁剪多余能力

如果保留的能力数量多于"新能力数量"，按以下优先级裁剪：

| 裁剪优先级 | 能力类型 | 说明 |
|-----------|---------|------|
| 最先裁剪 | 单方能力（仅主卡或仅副卡有） | 非核心能力优先移除 |
| 其次裁剪 | 共有能力 | 仍有保留概率 |
| 不可裁剪 | 必带能力 | 永远不会被移除 |

### 5.2.1 能力继承规则（按稀有度分级）

副卡的能力在融合时有概率被主卡继承，继承概率根据**能力的稀有度**分级：

| 能力稀有度 | 继承概率 | 说明 |
|-----------|---------|------|
| 普通能力（normal） | 30% | 常见能力，较容易继承 |
| 核心能力（core） | 20% | 稀有能力，较难继承 |
| 隐藏能力（hidden） | 10% | 极稀有能力，极难继承 |

**继承规则细节**：

1. **去重原则**：如果主卡已经拥有与副卡相同的能力，则该能力不会被重复添加（主卡的保留），但该能力会按"共有能力"规则以80%概率保留。
2. **槽位上限**：继承的能力数量不能超过当前品质的能力槽上限。如果继承后超出上限，按裁剪规则移除优先级最低的能力。
3. **独立判定**：副卡每个能力的继承判定是**独立**的，即可能出现副卡3个能力中2个被继承、1个没被继承的情况。
4. **主卡独有能力**：主卡独有的能力保留概率统一为50%，不受稀有度影响（因为主卡是"本体"，保留概率高于副卡继承概率）。

**示例**：

```
主卡能力：[厚甲(普通), 守护之盾(核心), 扎根(普通)]
副卡能力：[再生(普通), 暴击强化(核心), 猎杀本能(隐藏)]

融合判定过程：
- 厚甲：仅主卡有 → 50%保留（假设保留）
- 守护之盾：仅主卡有 → 50%保留（假设保留）
- 扎根：仅主卡有 → 50%保留（假设丢失）
- 再生：仅副卡有，普通能力 → 30%继承（假设继承）
- 暴击强化：仅副卡有，核心能力 → 20%继承（假设丢失）
- 猎杀本能：仅副卡有，隐藏能力 → 10%继承（假设丢失）

最终能力：[厚甲, 守护之盾, 再生]
```

### 5.3 隐藏融合能力

特定能力组合在融合时有概率产生**隐藏融合能力**——一个全新的能力，替换掉原来的两个能力。

#### 隐藏融合规则

```
如果主卡有能力A，副卡有能力B，且(A, B)在隐藏融合表中：
    有一定概率将A和B替换为隐藏能力C
```

#### 隐藏融合配置表（fusionAbilities.xlsx）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| fusionId | string | 唯一ID，如 "FUSION_001" |
| abilityA | string | 能力A的ID |
| abilityB | string | 能力B的ID |
| resultAbility | string | 融合产生的隐藏能力ID |
| probability | float | 触发概率（如 0.3 = 30%） |
| description | string | 融合描述（展示给玩家） |

#### 隐藏融合示例

| 能力A | 能力B | 隐藏结果 | 概率 | 描述 |
|-------|-------|---------|------|------|
| TANK_A_01 厚甲 | TANK_C_01 再生 | TANK_HIDDEN_01 不灭甲 | 30% | 厚甲与再生命力融合，获得持续减伤+回复的复合能力 |
| DPS_A_01 暴击强化 | DPS_B_01 穿甲 | DPS_HIDDEN_01 暴击穿透 | 30% | 暴击时无视目标减伤，暴击伤害额外+10% |
| SUP_A_01 持续恢复 | SUP_B_01 攻击鼓舞 | SUP_HIDDEN_01 战歌 | 30% | 治疗的同时为被治疗者提供攻击力加成 |

**设计意图**：
- 隐藏融合能力是"彩蛋"，鼓励玩家实验不同能力组合
- 隐藏能力通常比原始能力更强（1+1>2），但触发概率不高
- 隐藏能力也需要配入 abilities.xlsx，有完整的数值和描述

### 5.4 能力变化完整流程

```
1. 确定新能力数量 = 原数量 + random(-1, 0, +1)
2. 约束：不低于1，不超过品质槽上限
3. 必带能力 → 100%保留
4. 共有能力 → 80%保留
5. 主卡独有能力 → 50%保留
6. 副卡独有能力 → 按稀有度继承（普通30%/核心20%/隐藏10%）
7. 检查隐藏融合触发
8. 如果数量不足 → 从副卡能力/能力池补充
9. 如果数量过多 → 按优先级裁剪
10. 冲突检查 → 确保不违反冲突规则
11. 去重检查 → 确保不出现重复能力
12. 输出最终能力列表
```

## 六、技能处理规则

### 6.1 技能保留

融合后主卡的**技能列表和技能等级完全保留**，不受副卡影响。

| 项目 | 说明 |
|------|------|
| 技能ID | 不变 |
| 技能等级 | 不变（已升级的技能等级保留） |
| 技能倍率/冷却 | 不变 |
| 附加效果 | 不变 |

### 6.2 满觉醒卡的技能

如果主卡已满觉醒（3个技能全部6级），融合后技能仍然保持满觉醒状态。

## 七、融合界面与演出

### 7.1 融合流程

```
玩家进入融合界面
    ↓
选择主卡（从背包中选择）
    ↓
选择副卡（从同品质卡中选择，不可选主卡自身）
    ↓
预览融合结果（数值范围预览，能力变化预览，副卡能力继承概率预览）
    ↓
确认融合 → 消耗菌丝 + 副卡消失
    ↓
播放融合动画
    ↓
展示融合结果（数值、能力、品质变化）
```

### 7.2 融合预览

在确认融合前，玩家可以看到以下预览信息：

| 预览项 | 内容 |
|--------|------|
| 数值范围 | 新baseHp/baseAtk/baseSpd的最小值和最大值（基于0.2和0.8系数） |
| 能力变化 | 主副卡能力对比，标注"必保留"/"可能保留"/"可能丢失"/"继承概率" |
| 副卡能力继承概率 | 每个副卡能力的继承概率（普通30%/核心20%/隐藏10%） |
| 同元素加成提示 | 如果主副卡同元素，显示"品质提升概率+5%" |
| 品质提升概率 | 显示"20%概率品质提升"（同元素时显示"25%概率品质提升"） |
| 隐藏融合提示 | 如果主副卡有能力组合在隐藏融合表中，显示"可能触发隐藏融合" |
| 消耗 | 菌丝数量 |

### 7.3 融合动画演出

| 品质变化 | 动画 |
|---------|------|
| 品质不变 | 两张卡合为一体，光芒闪烁，数值和能力变化展示（约2秒） |
| 品质+1 | 两张卡合为一体，品质色光柱升起，卡框变化动画，能力槽扩展动画（约3-4秒） |
| 触发隐藏融合 | 额外播放特殊动画：两个能力图标碰撞融合为新图标，粒子特效（约1-2秒叠加） |

## 八、副卡选择策略指南

> 本节面向玩家，指导如何选择合适的副卡进行融合。副卡的选择是融合系统**最核心的策略决策**，直接影响融合结果。

### 8.1 副卡选择的核心原则

| 原则 | 说明 |
|------|------|
| 看数值 | 副卡的初始数值越高，融合后的期望数值越高 |
| 看能力 | 副卡的稀有能力有概率被继承，选对副卡可以获得珍贵能力 |
| 看元素 | 同元素副卡提供+5%品质提升概率加成 |
| 看隐藏组合 | 特定能力组合可能触发隐藏融合，产生强力新能力 |

### 8.2 数值维度：选择高初始值副卡

融合数值公式为 `(主卡初始值 + 副卡初始值) × 随机(0.2, 0.8)`，期望系数为0.5。

**策略推论**：
- 如果副卡某项初始值 > 主卡该项初始值，融合后期望值 > 主卡原始值（有收益）
- 如果副卡某项初始值 < 主卡该项初始值，融合后期望值 < 主卡原始值（有风险）
- 如果副卡某项初始值 = 主卡该项初始值，融合后期望值 = 主卡原始值（不亏不赚）

**建议**：优先选择**初始数值高于或接近主卡**的副卡，以降低数值下降的风险。

**示例**：

```
主卡"苔甲·叶青" SSR：baseHp=3000, baseAtk=800, baseSpd=70

副卡A"锈叶·苏蔓" SSR：baseHp=2800, baseAtk=1100, baseSpd=85
→ 期望结果：HP=2900(-3%), ATK=950(+19%), SPD=77.5(+11%)
→ 评价：攻击和速度有提升空间，HP略有风险，整体不错

副卡B"废铁·老陈" SSR：baseHp=3500, baseAtk=600, baseSpd=55
→ 期望结果：HP=3250(+8%), ATK=700(-13%), SPD=62.5(-11%)
→ 评价：HP有提升，但攻击和速度会下降，适合纯坦克定位

副卡C"烈焰·小红" SSR：baseHp=2200, baseAtk=1400, baseSpd=95
→ 期望结果：HP=2600(-13%), ATK=1100(+38%), SPD=82.5(+18%)
→ 评价：HP风险大，但攻击大幅提升，适合转为输出定位（高风险高回报）
```

### 8.3 能力维度：选择有价值的可继承能力

副卡的能力按稀有度有不同的继承概率，选择带有稀有能力的副卡可以在融合中获得强力能力。

**能力继承概率速查**：

| 副卡能力稀有度 | 继承概率 | 适合场景 |
|-------------|---------|---------|
| 普通能力 | 30% | 补充基础能力，概率较高 |
| 核心能力 | 20% | 获取稀有强力能力，概率中等 |
| 隐藏能力 | 10% | 获取极稀有能力，概率很低但收益极大 |

**策略建议**：
- 如果主卡缺少某个关键普通能力，选择拥有该能力的副卡（30%继承率，性价比高）
- 如果想获取核心能力，需要接受较低的概率（20%），可能需要多次融合尝试
- 隐藏能力（10%继承率）属于"彩票"级别，不建议专门为此选副卡，但如果副卡本身数值也不错则可以考虑

### 8.4 元素维度：同元素加成

| 条件 | 品质+1概率 | 说明 |
|------|-----------|------|
| 主副卡不同元素 | 20% | 基础概率 |
| 主副卡相同元素 | 25% | +5%加成 |

**策略建议**：
- 如果目标是品质提升，优先选择同元素副卡（25% vs 20%，提升25%相对概率）
- 如果目标是数值/能力优化，不必强求同元素，选择数值或能力更优的副卡更重要
- 同元素加成是锦上添花，不应成为副卡选择的唯一标准

### 8.5 隐藏融合维度：特定能力组合

如果主卡和副卡的能力组合在隐藏融合表中，有概率触发隐藏融合，产生1+1>2的强力新能力。

**策略建议**：
- 查阅社区攻略或游戏内提示，了解已发现的隐藏融合组合
- 如果发现有利可图的隐藏融合组合，可以专门选择对应能力的副卡
- 隐藏融合触发概率通常为20%-30%，属于中概率事件

### 8.6 综合选择优先级

| 优先级 | 考量因素 | 权重 |
|--------|---------|------|
| 1 | 副卡初始数值（越高越好） | 高 |
| 2 | 副卡可继承能力的价值 | 高 |
| 3 | 是否有隐藏融合组合 | 中 |
| 4 | 是否同元素（+5%品质概率） | 低 |

## 十、与技能升级系统的关系

### 10.1 满觉醒卡的融合

满觉醒卡（3个技能全部6级）可以作为主卡或副卡参与融合：

| 角色 | 说明 |
|------|------|
| 作为主卡 | 融合后技能等级保留（仍为满觉醒），但等级重置为1级 |
| 作为副卡 | 副卡的技能等级不影响融合结果（技能不参与融合计算） |

### 10.2 已品质+1的卡

已触发品质+1的卡：
- **仍可继续融合**（改变数值/能力），但不再有品质提升概率
- 可以作为副卡参与其他卡的融合

### 六、数值平衡要点

1. **期望值分析**：0.2~0.8的随机系数期望为0.5，即两张同品质卡初始值之和的一半。如果选择初始值相近的副卡，期望不亏不赚。但方差显著增大（标准差约为旧系统的2.5倍），单次融合的结果波动更大。
2. **品质+1的期望成本**：20%概率（同元素25%），平均需要4-5次融合才能升品质。以SR→SSR为例，期望消耗 = 5 × 8,000 = 40,000菌丝 + 5张SR副卡。
3. **隐藏融合的稀有度**：30%触发概率 × 需要特定能力组合 = 实际触发率较低，属于"彩蛋"性质。
4. **能力数量变化的期望**：-1/0/+1各33%，期望变化为0，长期来看能力数量稳定。
5. **副卡策略的平衡性**：高数值副卡通常意味着好用的卡牌，用作副卡有"消耗好卡"的代价，形成自然的策略权衡。
6. **同元素加成的平衡性**：+5%品质概率加成幅度适中，不会让同元素成为唯一选择，但提供了额外的策略维度。

### 一、需要新建的系统

#### FusionSystem.js — 主副卡融合系统

```javascript
class FusionSystem {
    constructor() {
        this.QUALITY_UP_CHANCE = 0.20; // 品质+1基础概率
        this.SAME_ELEMENT_BONUS = 0.05; // 同元素品质提升加成
        this.NUMBER_RANGE = { min: -1, max: 1 }; // 能力数量变化范围
        this.COEFFICIENT_RANGE = { min: 0.2, max: 0.8 }; // 数值系数范围（宽范围，高风险高回报）
        this.SHARED_ABILITY_KEEP_CHANCE = 0.80; // 共有能力保留概率
        this.MAIN_ONLY_ABILITY_KEEP_CHANCE = 0.50; // 主卡独有能力保留概率
        this.SUB_INHERIT_CHANCE = {
            normal: 0.30,  // 普通能力继承概率
            core: 0.20,    // 核心能力继承概率
            hidden: 0.10   // 隐藏能力继承概率
        };

        this.qualityOrder = ['N', 'R', 'SR', 'SSR', 'UR', 'LE'];
        this.fusionCosts = {
            N: 500, R: 2000, SR: 8000, SSR: 30000, UR: 100000
        };
    }

    /**
     * 执行融合
     * @param {string} mainCardInstanceId - 主卡实例ID
     * @param {string} subCardInstanceId - 副卡实例ID
     * @returns {Object} 融合结果
     */
    fuse(mainCardInstanceId, subCardInstanceId) {
        const mainCard = inventory.getCard(mainCardInstanceId);
        const subCard = inventory.getCard(subCardInstanceId);

        // 1. 验证
        this.validate(mainCard, subCard);

        // 2. 消耗菌丝
        const cost = this.fusionCosts[mainCard.quality];
        inventory.removeMycelium(cost);

        // 3. 消耗副卡
        inventory.removeCard(subCardInstanceId);

        // 4. 判定品质提升（含同元素加成）
        const isSameElement = mainCard.element === subCard.element;
        const qualityUpChance = mainCard.hasQualityUpgraded
            ? 0
            : this.QUALITY_UP_CHANCE + (isSameElement ? this.SAME_ELEMENT_BONUS : 0);
        const qualityUp = Math.random() < qualityUpChance;

        // 5. 计算新数值（三个属性独立随机）
        const mainConfig = ConfigManager.getCardById(mainCard.cardId);
        const subConfig = ConfigManager.getCardById(subCard.cardId);

        const hpCoeff = this.randomRange(this.COEFFICIENT_RANGE.min, this.COEFFICIENT_RANGE.max);
        const atkCoeff = this.randomRange(this.COEFFICIENT_RANGE.min, this.COEFFICIENT_RANGE.max);
        const spdCoeff = this.randomRange(this.COEFFICIENT_RANGE.min, this.COEFFICIENT_RANGE.max);

        const newBaseHp = Math.floor((mainConfig.baseHp + subConfig.baseHp) * hpCoeff);
        const newBaseAtk = Math.floor((mainConfig.baseAtk + subConfig.baseAtk) * atkCoeff);
        const newBaseSpd = Math.floor((mainConfig.baseSpd + subConfig.baseSpd) * spdCoeff);

        // 6. 判定能力变化（含按稀有度继承）
        const newAbilities = this.resolveAbilities(mainCard, subCard, qualityUp);

        // 7. 检查隐藏融合
        const hiddenFusion = this.checkHiddenFusion(mainCard.abilityIds, subCard.abilityIds);

        // 8. 应用变化
        const oldAbilityIds = [...mainCard.abilityIds];
        mainCard.baseHp = newBaseHp;
        mainCard.baseAtk = newBaseAtk;
        mainCard.baseSpd = newBaseSpd;
        mainCard.level = 1; // 等级重置为1
        mainCard.abilityIds = hiddenFusion ? hiddenFusion.newAbilities : newAbilities;
        mainCard.fusionCount = (mainCard.fusionCount || 0) + 1;

        let oldQuality = mainCard.quality;
        if (qualityUp) {
            mainCard.quality = this.qualityOrder[this.qualityOrder.indexOf(oldQuality) + 1];
            mainCard.hasQualityUpgraded = true;
        }

        // 9. 保存
        SaveSystem.save();

        return {
            success: true,
            qualityUp,
            isSameElement,
            oldQuality,
            newQuality: mainCard.quality,
            oldStats: { hp: mainConfig.baseHp, atk: mainConfig.baseAtk, spd: mainConfig.baseSpd },
            newStats: { hp: newBaseHp, atk: newBaseAtk, spd: newBaseSpd },
            coefficients: { hp: hpCoeff, atk: atkCoeff, spd: spdCoeff },
            oldAbilities: oldAbilityIds,
            newAbilities: mainCard.abilityIds,
            hiddenFusion: hiddenFusion || null,
            cost
        };
    }

    /**
     * 验证融合条件
     */
    validate(mainCard, subCard) {
        if (mainCard.quality === 'LE') {
            throw new Error('LE品质不可融合');
        }
        if (mainCard.quality !== subCard.quality) {
            throw new Error('主副卡品质必须相同');
        }
        const cost = this.fusionCosts[mainCard.quality];
        if (inventory.getMycelium() < cost) {
            throw new Error('菌丝不足');
        }
    }

    /**
     * 解析能力变化（含按稀有度继承规则）
     */
    resolveAbilities(mainCard, subCard, qualityUp) {
        const mainAbilities = mainCard.abilityIds;
        const subAbilities = subCard.abilityIds;
        const mainConfig = ConfigManager.getCardById(mainCard.cardId);

        // 1. 确定新能力数量
        const countChange = this.randomInt(this.NUMBER_RANGE.min, this.NUMBER_RANGE.max);
        let newCount = mainAbilities.length + countChange;
        const maxSlots = AbilityGenerator.getMaxSlots(qualityUp
            ? this.qualityOrder[this.qualityOrder.indexOf(mainCard.quality) + 1]
            : mainCard.quality);
        newCount = Math.max(1, Math.min(newCount, maxSlots));

        // 2. 分类能力
        const forcedAbilities = mainConfig.forcedAbilities || [];
        const sharedAbilities = mainAbilities.filter(a => subAbilities.includes(a));
        const mainOnlyAbilities = mainAbilities.filter(a =>
            !subAbilities.includes(a) && !forcedAbilities.includes(a)
        );
        const subOnlyAbilities = subAbilities.filter(a => !mainAbilities.includes(a));

        // 3. 保留判定
        const kept = [];

        // 必带能力100%保留
        for (const fa of forcedAbilities) {
            if (mainAbilities.includes(fa)) kept.push(fa);
        }

        // 共有能力80%保留
        for (const sa of sharedAbilities) {
            if (!kept.includes(sa) && Math.random() < this.SHARED_ABILITY_KEEP_CHANCE) {
                kept.push(sa);
            }
        }

        // 主卡独有能力50%保留
        for (const mo of mainOnlyAbilities) {
            if (!kept.includes(mo) && Math.random() < this.MAIN_ONLY_ABILITY_KEEP_CHANCE) {
                kept.push(mo);
            }
        }

        // 副卡独有能力 → 按稀有度继承
        for (const so of subOnlyAbilities) {
            if (kept.includes(so)) continue; // 去重：主卡已有则跳过
            const abilityConfig = ConfigManager.getAbilityById(so);
            const rarity = abilityConfig ? abilityConfig.rarity : 'normal';
            const inheritChance = this.SUB_INHERIT_CHANCE[rarity] || this.SUB_INHERIT_CHANCE.normal;
            if (Math.random() < inheritChance) {
                kept.push(so);
            }
        }

        // 4. 补充或裁剪
        while (kept.length < newCount) {
            // 从副卡未继承的能力中补充
            const remaining = subOnlyAbilities.filter(a => !kept.includes(a));
            if (remaining.length > 0) {
                kept.push(remaining[Math.floor(Math.random() * remaining.length)]);
            } else {
                // 从能力池中随机选取
                const pool = ConfigManager.getRandomAbilities(mainCard.quality);
                const available = pool.filter(a => !kept.includes(a) && !this.isConflicting(a, kept));
                if (available.length > 0) {
                    kept.push(AbilityGenerator.weightedRandom(available));
                } else {
                    break; // 无法再补充
                }
            }
        }

        while (kept.length > newCount) {
            // 裁剪非必带能力
            const removable = kept.filter(a => !forcedAbilities.includes(a));
            if (removable.length > 0) {
                const toRemove = removable[Math.floor(Math.random() * removable.length)];
                kept.splice(kept.indexOf(toRemove), 1);
            } else {
                break; // 无法再裁剪
            }
        }

        // 5. 冲突检查
        AbilityGenerator.checkConflicts(kept);

        // 6. 去重检查
        const unique = [...new Set(kept)];

        return unique;
    }

    /**
     * 检查隐藏融合
     */
    checkHiddenFusion(mainAbilityIds, subAbilityIds) {
        const allAbilities = [...mainAbilityIds, ...subAbilityIds];
        const fusionTable = ConfigManager.getAllFusionAbilities();

        for (const fusion of fusionTable) {
            if (allAbilities.includes(fusion.abilityA) && allAbilities.includes(fusion.abilityB)) {
                if (Math.random() < fusion.probability) {
                    // 触发隐藏融合
                    const newAbilities = [...mainAbilityIds];
                    // 移除A和B
                    const idxA = newAbilities.indexOf(fusion.abilityA);
                    const idxB = newAbilities.indexOf(fusion.abilityB);
                    if (idxA !== -1) newAbilities.splice(idxA, 1);
                    if (idxB !== -1) newAbilities.splice(newAbilities.indexOf(fusion.abilityB), 1);
                    // 添加结果能力
                    newAbilities.push(fusion.resultAbility);

                    return {
                        triggered: true,
                        abilityA: fusion.abilityA,
                        abilityB: fusion.abilityB,
                        resultAbility: fusion.resultAbility,
                        fusionId: fusion.fusionId,
                        newAbilities
                    };
                }
            }
        }

        return null;
    }

    /**
     * 生成融合预览
     */
    generatePreview(mainCardInstanceId, subCardInstanceId) {
        const mainCard = inventory.getCard(mainCardInstanceId);
        const subCard = inventory.getCard(subCardInstanceId);
        const mainConfig = ConfigManager.getCardById(mainCard.cardId);
        const subConfig = ConfigManager.getCardById(subCard.cardId);

        // 数值范围（基于0.2和0.8系数）
        const minHp = Math.floor((mainConfig.baseHp + subConfig.baseHp) * 0.2);
        const maxHp = Math.floor((mainConfig.baseHp + subConfig.baseHp) * 0.8);
        const minAtk = Math.floor((mainConfig.baseAtk + subConfig.baseAtk) * 0.2);
        const maxAtk = Math.floor((mainConfig.baseAtk + subConfig.baseAtk) * 0.8);
        const minSpd = Math.floor((mainConfig.baseSpd + subConfig.baseSpd) * 0.2);
        const maxSpd = Math.floor((mainConfig.baseSpd + subConfig.baseSpd) * 0.8);

        // 品质提升概率（含同元素加成）
        const isSameElement = mainCard.element === subCard.element;
        const canQualityUp = !mainCard.hasQualityUpgraded && mainCard.quality !== 'LE';
        const qualityUpChance = canQualityUp
            ? this.QUALITY_UP_CHANCE + (isSameElement ? this.SAME_ELEMENT_BONUS : 0)
            : 0;

        // 隐藏融合检查
        const possibleFusions = this.checkPossibleHiddenFusions(mainCard.abilityIds, subCard.abilityIds);

        // 能力变化预览（含副卡能力继承概率）
        const abilityPreview = this.previewAbilityChanges(mainCard, subCard);

        return {
            stats: {
                hp: { min: minHp, max: maxHp, current: mainConfig.baseHp },
                atk: { min: minAtk, max: maxAtk, current: mainConfig.baseAtk },
                spd: { min: minSpd, max: maxSpd, current: mainConfig.baseSpd }
            },
            isSameElement,
            qualityUpChance,
            possibleFusions,
            abilityPreview,
            cost: this.fusionCosts[mainCard.quality]
        };
    }

    /**
     * 预览能力变化（含副卡能力按稀有度的继承概率）
     */
    previewAbilityChanges(mainCard, subCard) {
        const mainConfig = ConfigManager.getCardById(mainCard.cardId);
        const forcedAbilities = mainConfig.forcedAbilities || [];
        const shared = mainCard.abilityIds.filter(a => subCard.abilityIds.includes(a));
        const mainOnly = mainCard.abilityIds.filter(a => !subCard.abilityIds.includes(a));
        const subOnly = subCard.abilityIds.filter(a => !mainCard.abilityIds.includes(a));

        // 副卡能力按稀有度标注继承概率
        const subOnlyWithRarity = subOnly.map(id => {
            const abilityConfig = ConfigManager.getAbilityById(id);
            const rarity = abilityConfig ? abilityConfig.rarity : 'normal';
            const chance = this.SUB_INHERIT_CHANCE[rarity] || this.SUB_INHERIT_CHANCE.normal;
            const rarityLabel = { normal: '普通', core: '核心', hidden: '隐藏' }[rarity] || '普通';
            return {
                id,
                status: 'new',
                label: `${rarityLabel}，${Math.round(chance * 100)}%继承`,
                chance,
                rarity
            };
        });

        return {
            forced: forcedAbilities.map(id => ({ id, status: 'guaranteed', label: '必保留' })),
            shared: shared.map(id => ({ id, status: 'likely', label: '80%保留', chance: 0.8 })),
            mainOnly: mainOnly.map(id => ({ id, status: 'risky', label: '50%保留', chance: 0.5 })),
            subOnly: subOnlyWithRarity
        };
    }

    /**
     * 检查可能的隐藏融合
     */
    checkPossibleHiddenFusions(mainAbilityIds, subAbilityIds) {
        const allAbilities = [...mainAbilityIds, ...subAbilityIds];
        const fusionTable = ConfigManager.getAllFusionAbilities();
        const possible = [];

        for (const fusion of fusionTable) {
            if (allAbilities.includes(fusion.abilityA) && allAbilities.includes(fusion.abilityB)) {
                possible.push({
                    fusionId: fusion.fusionId,
                    abilityA: fusion.abilityA,
                    abilityB: fusion.abilityB,
                    resultAbility: fusion.resultAbility,
                    probability: fusion.probability,
                    description: fusion.description
                });
            }
        }

        return possible;
    }

    // 工具方法
    randomRange(min, max) {
        return min + Math.random() * (max - min);
    }

    randomInt(min, max) {
        return Math.floor(Math.random() * (max - min + 1)) + min;
    }

    isConflicting(abilityId, existingIds) {
        return AbilityGenerator.isConflicting(abilityId, existingIds);
    }
}
```

### 六、关键实现注意事项

1. **随机系数独立生成**：HP、ATK、SPD三个属性的随机系数必须独立生成，不能共用同一个系数值。
2. **副卡能力稀有度查询**：ConfigManager 需要支持通过 abilityId 查询能力的 rarity 字段，用于计算继承概率。
3. **同元素判定**：品质提升概率计算时需要先判定主副卡是否同元素，再加成5%。
4. **去重处理**：能力继承后需要去重，避免同一能力出现多次。
5. **改造晶片完全解耦**：融合函数 fuse() 不接受任何晶片参数，融合流程中不涉及晶片逻辑。
6. **预览准确性**：融合预览中的数值范围应基于0.2和0.8系数计算，能力继承概率应按稀有度准确显示。
